# 🚢 AI开发船队 — 使用指南

## 这套系统解决什么问题？

当你给AI一个需求时，AI最常见的坏习惯是：**不问你任何问题，直接给你一堆代码或方案**。结果呢？大方向可能偏了，细节大概率不是你想要的，来回返工浪费大量时间。

这套提示词工程强制AI在动手前必须：
1. **复述理解** — 用自己话说一遍需求，让你确认它没理解错
2. **追问澄清** — 把模糊点列出来，用选择题的方式让你快速决策
3. **亮出方案** — 给出多套方案并说明利弊，让你选而不是替你定
4. **分段推进** — 每个阶段做完要你确认，才进入下一阶段

## 船队角色一览

| 角色 | 文件 | 什么时候用 |
|------|------|-----------|
| 📋 总指挥 | `总控提示词.md` | 项目启动时，把需求丢给它，它会规划全流程 |
| 🎯 产品经理 | `产品经理.md` | 需求不清晰时，帮你把模糊想法变成明确的需求文档 |
| 🏗️ 架构师 | `架构师.md` | 需求明确了，需要做技术方案选型和架构设计 |
| 🔧 技术负责人 | `技术负责人.md` | 架构定了，需要拆任务、排期、写开发计划 |
| 🔍 代码审计 | `代码审计.md` | 代码写完了，需要审查质量和安全性 |
| ✅ 交付验收 | `交付验收.md` | 功能做完了，需要按标准逐条验收 |

## 使用流程

### 方式一：新项目（推荐）

直接把 `总控提示词.md` 的内容发给AI，然后告诉它你的项目想法。总指挥会引导你走完全流程。

```
1. 发送「总控提示词」给AI
2. 告诉AI你的项目想法
3. AI开始提问，你逐个回答
4. AI输出项目全景，你确认后进入下一阶段
5. 按AI指引，依次调用对应角色
```

### 方式二：单阶段使用

如果你只有某个阶段需要帮助，直接用对应角色的提示词。

```
1. 打开对应角色的 .md 文件
2. 填入【用户填写】部分
3. 把完整提示词发给AI
4. 按AI提问交互，直至输出结果
```

## 核心原则（每个角色都遵循）

### 🛑 三不原则
- **不猜测**：不确定的地方必须问，不允许「我觉得用户可能是这个意思」
- **不跳步**：严格按照工作流程，禁止跳过检查点
- **不替用户做决策**：涉及取舍的地方必须列出选项让用户选

### ✅ 三必原则
- **必复述**：收到需求后必须用自己的话复述一遍
- **必追问**：模糊点必须列出具体问题追问
- **必确认**：每个阶段输出后必须等用户说「确认」才能继续

## 填写需求参考

如果你还没想清楚，可以按这个框架来组织你的想法：

```
1. 背景：为什么要做这个？现在有什么痛点？
2. 目标：做完后要达到什么效果？可衡量的指标是什么？
3. 用户：谁会用？他们有什么特征？
4. 约束：时间、预算、技术栈、平台有什么限制？
5. 交付标准：做到什么程度算「完成」？
6. 不做什么：明确划定边界，什么不在本次范围内？
```

## 常见用法示例

```
# 示例1：有个网站想法但不确定怎么做
→ 用「总控提示词」+ 你的想法

# 示例2：已有的功能要重构
→ 用「架构师」+ 现状描述 + 重构目标

# 示例3：代码写完了想检查质量
→ 用「代码审计」+ 贴代码或指定文件

# 示例4：功能上线前最后检查
→ 用「交付验收」+ 需求文档 + 当前功能描述
```
